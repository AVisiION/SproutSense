# SproutSense: Smart Agriculture Documentation

## 1. Project Overview and Problem Definition

### 1.1 Project Name and Tagline
**Project Name:** SproutSense  
**Tagline:** "Empowering Farmers with Data-Driven Growth."

### 1.2 Real-World Problem
Agriculture in developing countries faces multifaceted challenges impacting both immediate yields and long-term sustainability:
*   **Water Inefficiency:** Unscientific water management and flood irrigation lead to millions of liters of wasted water daily.
*   **Unpredictable Microclimates:** Hyper-local variations in temperature and humidity leave crops highly vulnerable to stress periods, which often go unnoticed until physical damage occurs.
*   **Lack of Real-Time Data:** Institutional knowledge and intuition are insufficient against rapidly changing climate conditions. Small and marginal farmers operate "in the blind," lacking localized metrics.
*   **Late Disease Detection:** Leaf diseases spread rapidly in dense fields. Relying on visual human inspection often detects pathogenic outbreaks only after catastrophic yield reduction.

### 1.3 Objectives and Scope
**Technical Objectives:**
*   Develop a dual-microcontroller edge layer utilizing an ESP32 for continuous environmental telemetry and an ESP32-CAM for on-demand leaf image capture and localized disease inference.
*   Build a scalable MERN (MongoDB, Express, React, Node.js) web architecture capable of real-time bidirectional communication via WebSockets.
*   Implement hybrid AI logic: basic threshold-based heuristics on the cloud for immediate intervention, and integration of lightweight ML classification for disease risk.
*   Secure the entire system using Role-Based Access Control (RBAC) governed by JWT tokens.

**Social Objectives:**
*   Eliminate the technological entry barrier for marginal farmers by providing a localized, visual-first intuitive interface.
*   Maximize crop yields while minimizing ecological degradation (water table depletion, fertilizer runoff).

**In-Scope Features:**
*   Continuous telemetry recording (Moisture, Temp, Humidity, Light).
*   Dual ESP32 operations communicating securely with a remote Node.js backend.
*   Real-time MERN Web Dashboard deployed on Netlify (frontend) and Render (backend).
*   Configurable AI-driven alerts for immediate stress and irrigation.

**Out-of-Scope Features:**
*   Complex mechanical automation (e.g., autonomous tractors or drone pesticide spraying).
*   Heavyweight, high-cost 5G/satellite subscriptions requiring specialized, expensive licensing.
*   Direct API integration with national commodity trade markets.

### 1.4 Target Users and Stakeholders
**Primary Users:**
*   **Small/Marginal Farmers:** Need highly simplified, actionable prompts ("Water Now", "Possible Infection"). Emphasizes SMS or bold push-notifications.
*   **Progressive Farm Managers:** Require deeper metrics, historical graphs (Recharts integration), and granular sensor tuning logic.
*   **Agricultural Extension Officers:** Gain access to aggregate, anonymized "bird's eye" views of multi-farm clusters to predict regional outbreak trajectories.

**Secondary Stakeholders:**
*   **Government & Environment Sector:** Benefit immensely from the reduction in water pumping frequency and precise tracking of localized climate data.

## 3. System Architecture and Structure

### 3.1 High-Level Block Diagram (Text Description)
The platform rests on a modern hybrid edge-cloud architecture:
1.  **Sensor/Field Layer:** 
    *   *Telemetry End:* Capacitive moisture probe, DHT22 (temp/humidity), and distinct LDR photoresistor taking continuous physical measurements.
    *   *Vision End:* An OV2640 camera module observing canopy geometry and leaf health.
2.  **Communication Layer:** Local 2.4GHz Wi-Fi interfacing with standard TCP/IP. Uses HTTP REST protocols for bulk data blocks and Websockets for sub-second real-time gauge updates.
3.  **Edge/Controller Layer:** Dual microcontrollers:
    *   *ESP32-SENSOR:* Processes continuous analog-to-digital conversions and packages JSON arrays.
    *   *ESP32-CAM:* Triggers only when requested, captures a JPEG buffer, and streams/posts it to the server.
4.  **Cloud/Server Layer (MERN Backend):** Node.js Express server orchestrates all activity. MongoDB Atlas persistently logs telemetry to Time-Series collections.
5.  **Application Layer:** React SPA application presenting customized interactive data to registered users based on explicit user-role tiers.

### 3.2 Detailed Components
*   **Hardware:**
    *   `NodeMCU ESP32S WROOM-32`: Main telemetry logic controller.
    *   `ESP32-CAM (AI-Thinker)`: Specialized vision processing edge device.
    *   `Sensors`: DHT22 (Digital Temp/Hum), Capacitive Soil Moisture v1.2, basic LDR module.
*   **Backend Software (apps/api):**
    *   `Express.js`: Routing and middleware controller.
    *   `Mongoose`: Strict schema modeling for MongoDB.
    *   `ws`: Native WebSocket protocol for real-time pushes.
    *   `Bcrypt & JsonWebToken`: Cryptographic security processing.
*   **Frontend Software (apps/web):**
    *   `React & Vite`: Next-generation frontend tooling providing rapid HMR and optimized production bundles.
    *   `Tailwind CSS`: Utility-first semantic styling preventing massive CSS payload bottlenecks.
    *   `Recharts`: Dedicated performant charting graphics.

### 3.3 Data Flow and Sequence (Step-by-Step)
1.  **Sensing (Periodic Wake):** The ESP32 wakes from light sleep, triggers the DHT22 digital pull, and reads analog voltage offsets from the soil probe.
2.  **Transmission (JSON POST):** The ESP32 formats an HTTP POST containing `{ "deviceId": "NODE_12", "timestamp": "...", "metrics": {"moisture": 42, "temperature": 31.5} }` to `/api/sensors/ingest`.
3.  **Backend Processing:**
    *   API Middleware verifies device authentication tokens.
    *   The packet is inserted into the MongoDB Atlas `SensorData` time-series collection.
    *   The `AiController` script intercepts the raw parameters.
4.  **Heuristic Evaluation:** The AI logic compares incoming 42% moisture against the plant's defined optimal threshold (e.g., Target: 60%). 
5.  **WebSocket Push:** The backend emits a live Socket event `UPDATE_NODE_12`. The React frontend instantly redraws the circular moisture gauge from green to yellow, warning "Action Needed."

### 3.4 Connections and Interfaces (Reference for Humans and AI)
*   **Hardware → Sensor:** `ESP32_PIN_34` (ADC) ↔ `Analog_Out_Moisture`. 
*   **Hardware → Camera:** Native FPC ribbon connection holding the `OV2640` lens to the ESP32-CAM board.
*   **Edge → Cloud API:** 
    *   **Protocol:** HTTP POST / GET, WebSocket `ws://` or `wss://`.
    *   **Auth:** Bearer Token via Header `Authorization: Bearer <token>`.
*   **Cloud API → MongoDB:** `mongoose.connect()` utilizing the hidden `MONGODB_URI` environment variable, connecting to `cluster0.mongodb.net`.
*   **Frontend → Cloud API:** 
    *   Axios instance mapped to `VITE_API_BASE_URL`.
    *   Endpoints: `/api/auth/login`, `/api/sensors/history`, `/api/ai/predict`.

## 4. Hardware Design (Student-Friendly)

### 4.1 Hardware List
| Identifier | Component | Role / Specification |
| :--- | :--- | :--- |
| `MCU_MAIN` | ESP32 WROOM-32 | Central intelligence for telemetry and networking. |
| `MCU_VISION` | ESP32-CAM AI-Thinker | Independent processor for handling massive image arrays. |
| `SENS_MOIST` | Capacitive Soil Moisture v1.2 | Corrosion-resistant dielectric soil measurement. |
| `SENS_ATMOS` | DHT22 Module | Highly accurate environmental Temp/Hum metrics. |
| `SENS_LDR` | Light Dependent Resistor | Detects photo-cycles and excessive light exposure. |
| `PWR_SUPPLY` | 5V 10000mAh Power Bank | Ensures high-uptime off-grid performance for demonstrations. |

### 4.2 Justification for Choices
*   **Good Option (Current):** Using two separate ESP32s prevents memory/stack exhaustion. Combining heavy image processing and constant Wi-Fi telemetry polling on a single ESP32 causes thermal throttling and crashes. Splitting the architecture into `MCU_MAIN` and `MCU_VISION` guarantees resilience.
*   **Best Option (Enterprise):** Using extreme low-power LoRa modules for telemetry connected to a central base station, combined with 4G solar-powered heavy edge servers for ML capability. 
*   **Student Priority:** Reliability. Students must prioritize components that compile easily repeatedly over the Arduino IDE without heavy SDK debugging, making the standard ESP32 irreplaceable.

### 4.3 Physical Layout and Deployment
*   **Soil Placement:** Insert the `SENS_MOIST` blade straight down near the core root bundle, never directly against the stalk to avoid mechanical damage to the plant.
*   **Atmospheric Placement:** Maintain the `SENS_ATMOS` (DHT22) under a small 3D-printed or plastic cowl to prevent direct sunlight radiation or rain droplets skewing thermal/humidity accuracy.
*   **Enclosure:** Main `MCU_MAIN` and wires must be housed within a sealed Tupperware-style container fitted with rubber cable-glands, protecting delicate jumper wires against field rodents and immediate rain.

## 5. Software, AI, and Data

### 5.1 Software Stack Overview
The MERN + ESP C++ structure is the undisputed champion for student IoT, allowing total JavaScript unification on the web side while utilizing native low-level memory handling on the edge.
*   `C++ / Arduino`: Direct memory manipulation over I/O pins.
*   `Node.js`: Unmatched capability managing simultaneous asynchronous non-blocking API calls.
*   `MongoDB`: Documents inherently match incoming JSON sensor structures natively, removing the need for strict relational SQL migrations on the fly.
*   `React.js`: The frontend renders individual sensor readouts as distinct, encapsulated functional components preserving the DOM structure.

### 5.2 Data Model and Storage
*   **Collections:**
    *   `User`: `_id`, `email`, `passwordHash`, `role` (Admin/Farmer), `assignedDevices[]`.
    *   `Device`: `hardwareId`, `status`, `location`, `plantType`.
    *   `Telemetry`: `timestamp`, `deviceId`, `{moisture, temp, hum, light}`. (Configured as a MongoDB Native Time-Series Collection for massive indexing speed).
    *   `Alerts`: `severity`, `resolved`, `message_text`.

### 5.3 AI / ML Logic (High-Level)
*   **Phase 1: Telemetry Heuristics:** The backend uses an expert rules engine. If `delta(Moisture)` falls at a pace > `10% per hour` and `Temperature` > `34C`, the engine fires a preemptive `High Stress Alert` before the plant visually wilts.
*   **Phase 2: Vision Classification:** The ESP32-CAM captures an image buffer. The backend (or an integrated python microservice if added) runs a tiny, pre-quantized Convolutional Neural Network (MobileNet V2 / TensorFlow Lite) trained strictly on "Healthy Leaf" vs "Blight Infection". The output yields a percentage: `Fungal Risk: 83%`.

### 5.4 Integration and APIs
1.  **Authentication:** `POST /api/auth/login` → returns JWT. This JWT is appended to all subsequent calls.
2.  **Edge Ingestion:** `POST /api/sensors` → Requires an Edge-secret headers, guaranteeing external bots cannot flood fake farm data.
3.  **Farmer Retrieval:** `GET /api/sensors/devices/:id/current` → Serves dashboard rendering data.
4.  **AI Interaction:** `POST /api/ai/diagnose-image` (multipart/form-data containing the ESP32 JPEG) → returns a JSON diagnostics array.

## 9. Detailed Usage Guide and User Manual

### 9.1 Getting Started
1.  **Assembly Check:** Open the enclosure to ensure all pin connections (especially I2C/ADC pins) are tightly seated. Do not modify the screw-terminals unless instructed.
2.  **Power On Sequencing:** Connect the 5V power lines. Wait exactly 15 seconds. Observe the blue embedded LED on the ESP32.
    *   *Solid Blue:* Wi-Fi Connected.
    *   *Flashing Blue:* Searching/Booting.
3.  **Software Initiation:** Open the assigned Netlify domain link. Enter the administrative credentials provided during the backend `.env` setup.

### 9.2 Daily Operation
*   The dashboard default layer is the **Overview Module**.
*   **Status Ring:** Look instantly for standard colors. 
    *   Green: Health optimal.
    *   Orange: Moderate stress building, flag for checkup.
    *   Red: Severe stress. Immediate physical inspection required.
*   **AI Recommendations:** Scroll to the "AI Diagnostics" panel at the bottom. This translates recent fluctuations into human sentences (e.g., "Humidity has been above 80% for 12 hours. Ensure proper ventilation.").

### 9.3 Troubleshooting and FAQs
*   **Device is Offline (Red icon on Dashboard):**
    *   Check physical power bank delivery.
    *   Check Wi-Fi router proximity (Ensure distance is < 15 meters without thick concrete).
*   **Moisture reading locked at 100% or 0% constantly:**
    *   The Capacitive sensor has been unplugged from the internal ADC port, or the sensor has fully oxidized/failed physically. Clean the probe with a dry cloth and reinsert.
*   **ESP32-CAM images are distorted/missing:**
    *   Confirm the tiny FPC lens connector hasn't slipped out during transit. Maintain higher power (min 2A output from power supply) when triggering the camera.

## 12. Test Plan, Evaluation, and Limitations

### 12.1 Testing Strategy
*   **Firmware Stress Test:** Artificially reboot Wi-Fi routers randomly. Ensure the ESP32 auto-reestablishes standard HTTP connections successfully over 20 concurrent drops.
*   **API Load/Security Testing:** Ensure unauthorized GET requests to `/api/sensors` strictly return `HTTP 401 Unauthorized`. Use tools like Postman to run functional endpoint testing.
*   **Frontend End-to-End (E2E):** Trigger artificial JSON payloads imitating a massive temperature spike. Verify the React DOM flashes red and dispatches warnings within 1-2 seconds over the WebSocket stream.

### 12.2 Metrics and Evaluation
*   **Response Latency:** Cloud ingestion to frontend render must remain strictly under a 2-second median.
*   **Hardware Accuracy:** DHT22 temperature readings must calibrate closely (+/- 0.5 degrees) alongside physical mercury thermometers.

### 12.3 Limitations and Future Work
**Limitations:**
*   Reliance on central generic Wi-Fi protocols means the edge device cannot wander beyond the ~50-meter perimeter of standard rural hotspots.
*   AI constraints: A rigid "rule-engine" heuristic assumes perfect environmental truth. If a sensor degrades and reports 100°C erroneously, the engine will trigger a false positive without an advanced anomaly-detection pipeline.
*   Power density requires human intervention to swap or monitor power-bank levels regularly.

**Future Enhancements:**
*   Implement real-time weather API integration (e.g., OpenWeatherMap) directly into the Node.js backend. If it's scheduled to rain heavily in 2 hours, the backend automatically suppresses the "Irrigate Now" local alerts.
*   Transition from heavy 5V battery banks to efficient 3.3V 18650 lithium cells bonded to micro-solar chargers for infinite physical deployment uptime.
*   Deploy robust multi-lingual automated Voice-prompts (via Web Speech API or Twilio integration) converting visual dashboard data into local-language audio for totally illiterate end-users.
